Projet EGNARO - Hebergement d'un site web

Sept. 2024 - Jan. 2025

Contexte

C'est un projet qu'on a fait en cours. Le but c'etait d'heberger un site web sur un serveur Debian, avec Apache et MySQL. On devait tout configurer nous-memes, de l'installation du serveur jusqu'a la mise en ligne du site.

Ce que j'ai fait

  • Installation et configuration d'Apache sur Debian
  • Creation de la base de donnees MySQL et des tables
  • D'abord transfert par FTP, puis on est passe sur Git
  • Mise en place des sauvegardes automatiques de la base de donnees avec mysqldump (script cron quotidien)
  • J'ai aussi redige la doc d'installation pour que d'autres puissent refaire la manip

Technologies

Apache 2.4 MySQL 8 PHP 8.1 FileZilla Git Debian

Probleme rencontre

Apres avoir deploye le site sur le serveur, j'avais une erreur 403 Forbidden. J'ai mis du temps a comprendre que le probleme venait des permissions du fichier .htaccess et aussi de la directive AllowOverride qui etait desactivee dans la config Apache.

Ce que j'ai fait : j'ai corrige les permissions avec chmod 644 .htaccess et j'ai active AllowOverride All dans le VirtualHost. Apres ca, le site s'affichait normalement.

Mise en production

Voici les etapes qu'on a suivies pour mettre le site en ligne :

  1. Preparation du serveur - Installation d'Apache, PHP et MySQL sur la machine Debian. Configuration du VirtualHost.
  2. Transfert des fichiers - Au debut avec FileZilla, puis on a mis en place un depot Git pour faciliter les mises a jour.
  3. Base de donnees - Import du fichier SQL, creation d'un utilisateur dedie avec des droits limites.
  4. Tests - On a verifie que toutes les pages fonctionnent, que les formulaires envoient bien les donnees, etc.
  5. Documentation - J'ai fait un petit guide d'installation pour qu'un autre etudiant puisse refaire la meme chose.

Technologies

Probleme rencontre

Le site etait lent a charger, le score Lighthouse etait a 62/100. Apres avoir regarde ce qui posait probleme, c'etait les images. Elles etaient en PNG haute resolution, sans aucune compression.

Ce que j'ai fait : j'ai converti les images en WebP (format plus leger), j'ai ajoute loading="lazy" pour charger les images seulement quand on scroll dessus, et j'ai mis les attributs width et height pour eviter les sauts de page. Le score est passe a 94/100.

Course aux Salles - Application mobile jeu de piste en lycee

Sept. 2025 - Jan. 2026

Contexte

Projet scolaire en equipe de 4, realise en methode agile (4 sprints de 2 semaines). L'objectif etait de creer un jeu mobile multi-plateforme (Android/iOS) pour le lycee : des classes s'affrontent dans des epreuves geolocalises via QR/NFC, avec elimination progressive. J'etais charge de la partie professeur/admin : a la fois le back-end Symfony et l'interface web d'administration.

Ce que j'ai fait

  • Developpement de l'interface web admin (Symfony 7 + EasyAdmin) pour les professeurs
  • Creation et gestion des tournois, manches, zones et QR codes depuis l'admin
  • Suivi en direct des scores et eliminations via WebSockets (Mercure)
  • Commandes admin : Pause / Reprise / Ajustement du chrono en live
  • Generation et export du rapport PDF final pour l'equipe pedagogique
  • Participation aux ceremonies agiles : daily Discord, retrospectives de sprint

Technologies

Symfony 7 API Platform Mercure (WebSocket) PostgreSQL .NET MAUI GitHub Actions Trello Postman

Probleme rencontre

La synchronisation du chronometre entre le serveur et les clients mobiles etait difficile : chaque telephone avait une heure legrement differente, ce qui causait des decalages visibles sur l'ecran de compte a rebours.

Ce que j'ai fait : j'ai fait du serveur l'autorite de temps unique. A chaque connexion, le client recoit l'heure serveur et calcule un delta. Toutes les 10 secondes, le client se re-synchronise via Mercure pour corriger la derive. Le decalage est tombe sous 200ms, imperceptible pour les joueurs.

Organisation du projet (methode agile)

  1. Sprint 1 - Setup API Symfony, auth, connexion classe, scan QR simple, compte a rebours lobby.
  2. Sprint 2 - Enigmes (2 types), validation NFC, chrono epreuve avec fin automatique.
  3. Sprint 3 - Mini-jeux, elimination joueurs, panneau admin web (EasyAdmin), scoreboard temps reel.
  4. Sprint 4 - Tests de charge, securite, deploiement APK, rapport PDF final.

Projet Encheres - Plateforme d'encheres Symfony + Vue.js

Jan. - Mai 2026

Contexte

Projet scolaire realise en equipe de 2 sur 7 semaines (140h). L'objectif etait de concevoir une plateforme d'encheres complete avec une architecture separee : un backend API Symfony et un frontend Vue.js communiquant via Fetch. J'etais charge de la partie backend (API REST, authentification, logique metier).

Ce que j'ai fait

  • Conception du MCD et des entites Doctrine (User, Auction, Bid, Transaction)
  • Mise en place de l'authentification par token Bearer maison
  • Developpement des routes API REST (register, login, CRUD encheres, enchérir)
  • Gestion des regles metier : validation des encheres, clôture automatique
  • Creation d'une commande Symfony pour detecter et cloturer les encheres terminees

Technologies

Symfony 7 PHP 8 Doctrine ORM MySQL Vue.js 3 Fetch API Git

Probleme rencontre

Quand deux utilisateurs enchérissaient en meme temps, il pouvait arriver que le prix courant soit lu en meme temps par les deux, et les deux encheres soient acceptees alors que l'une devait etre refusee.

Ce que j'ai fait : j'ai utilise une transaction de base de donnees avec verrouillage de ligne (SELECT ... FOR UPDATE) dans le BidService. Si le prix avait change entre la lecture et l'ecriture, l'API renvoyait une erreur 409 Conflict pour indiquer que l'enchere n'etait plus valide.

Architecture et deploiement

L'application suit une architecture client/serveur stricte :

  1. Backend Symfony - API REST stateless, aucune session. Chaque requete est authentifiee via le header Authorization: Bearer TOKEN.
  2. Token maison - Genere avec bin2hex(random_bytes(32)) au login, stocke en base, verifie par un ApiTokenAuthenticator.
  3. Frontend Vue.js - SPA avec Vue Router, le token est stocke en localStorage et envoye dans chaque requete Fetch.
  4. Clôture automatique - Commande Symfony CloseAuctionsCommand qui detecte les encheres dont la end_date est depassee, determine le gagnant et cree une Transaction.

Exemple concret : IntersectionObserver

En cherchant comment animer des elements au scroll, je suis tombe sur l'API IntersectionObserver via un article sur MDN et une video de Kevin Powell.

Avant, j'aurais utilise un ecouteur scroll avec des calculs de position, mais c'est pas performant parce que ca se declenche a chaque pixel de scroll. IntersectionObserver est beaucoup mieux parce que c'est le navigateur qui gere la detection.

Je l'ai utilise dans ce portfolio pour faire apparaitre les sections au fur et a mesure qu'on descend dans la page.
